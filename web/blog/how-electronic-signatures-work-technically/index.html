How electronic signatures actually work, technically

May 28, 2026

Most people sign electronically without ever wondering what happens between 'click sign' and 'document received.' It's worth knowing.

When you sign a document electronically, six things happen in the background: identity check, document fingerprinting, signature capture, event logging, file modification, and delivery. Understanding the stack helps you evaluate whether the platform you're using produces records that will hold up.

Identity verification: the magic link

Modern e-signature platforms send a unique URL to each signer's email. That URL contains a one-time token that ties the signing session to the signer. The token is random (typically 32 bytes of cryptographic-grade entropy) and stored server-side so the platform can verify it on access. When the signer clicks the link, the platform records their IP address and timestamp before showing them the document. This becomes the 'signer identity' part of the audit record.

Document fingerprinting: SHA-256

Before any modification, the platform computes a SHA-256 hash of the original document bytes. A SHA-256 hash is a 64-character hexadecimal string. Change one byte of the document and the hash changes completely. The platform stores this hash in the audit certificate. Years later, anyone can verify 'yes, this is the document that was actually signed' by re-hashing the original PDF and comparing to the recorded hash.

Event logging: the audit trail

Every signing event is logged: invite sent, link clicked, document viewed, field filled, signature drawn, doc submitted. Each event records its timestamp, IP address, user agent, and event type. Together these form the chain of custody — the documentary evidence that the signing took place the way the platform says it did. When you receive a signed document, you should receive the audit certificate alongside it.
